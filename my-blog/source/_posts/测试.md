## 一、软件测试的四个阶段

软件测试通常分为以下四个阶段，各阶段的测试范围和关注点依次递进：

| 阶段 | 测试类型 | 说明 |
|:---:|:---:|------|
| **单元测试** | 白盒测试 | 对最小的可测试单元（方法/函数）进行验证 |
| **集成测试** | 灰盒测试 | 模块之间接口和交互的测试 |
| **系统测试** | 黑盒测试 | 整个系统功能与性能的全面验证 |
| **验收测试** | 黑盒测试 | 用户确认系统是否满足业务需求 |

### 各阶段详细说明

1. **单元测试（白盒）**：开发人员编写，基于代码内部逻辑，关注每个方法的正确性。
2. **集成测试（灰盒）**：介于黑白盒之间，既关注接口也关注部分内部实现，验证模块协作是否正常。
3. **系统测试（黑盒）**：测试人员执行，不考虑内部实现，从用户角度验证整体功能。
4. **验收测试（黑盒）**：最终用户或代表参与，确认系统是否符合预期需求。

## 二、为什么不用 main 方法做单元测试？

使用 `main` 方法进行测试存在以下问题：

| 问题 | 具体表现 |
|:---|:---|
| **代码耦合** | 测试代码与源代码未分离，混在一起难以维护 |
| **相互影响 | 一个测试方法失败会阻塞后续方法的执行 |
| **缺乏自动化** | 无法自动运行测试并生成规范的测试报告 |

---

## 三、JUnit 单元测试框架

JUnit 是 Java 生态中最常用的单元测试框架。

### 使用 JUnit 的好处

| 优势 | 说明 |
|:---|:---|
| **代码分离** | 测试类与源码独立存放（如 `src/test/java`），结构清晰便于维护 |
| **自动化测试** | 支持按需批量运行测试用例，无需手动逐个调用 |
| **结果分析** | 自动统计通过/失败/跳过的用例，生成可视化测试报告 |

### 对比总结

```
┌─────────────────────────┐     ┌─────────────────────────┐
│     main 方法测试        │     │    JUnit 单元测试        │
├─────────────────────────┤     ├─────────────────────────┤
│ ❌ 代码混在一起          │     │ ✅ 测试与源码分离        │
│ ❌ 一个失败全部停        │     │ ✅ 用例独立互不影响      │
│ ❌ 无报告产出            │     │ ✅ 自动生成测试报告      │
└─────────────────────────┘     └─────────────────────────┘
```
## 四、使用 JUnit 的步骤

1. 创建测试类，继承 `junit.framework.TestCase` 类。
2. 创建测试方法，方法名以 `test` 开头。
3. 使用 `@Test` 注解标记测试方法。
4. 运行测试用例，生成测试报告。

### 代码示例

在pom.xml中添加依赖:

```xml
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.9.1</version>
</dependency>
```

在XxxxTest.java中编写测试代码:

```java
@Test
//必须public void开头
public void testAdd() {
    int result = Calculator.add(1, 2);
    assertEquals(3, result);
}
```

## 五、断言（Assertion）—— 验证程序正确性的核心

### 为什么测试变绿不代表程序正确？

JUnit 运行测试方法时，如果**没有抛出异常**就会显示绿色 ✅。但以下情况即使结果错误也会变绿：

```java
// ❌ 没有断言 —— 即使逻辑错误，测试依然显示通过（绿色）
@Test
public void testAddWithoutAssertion() {
    int result = Calculator.add(1, 2); // 假设返回了错误的 100
    // 没有任何验证！测试永远绿色，但结果可能是错的
}
```

```
┌──────────────────────────────────────────────┐
│  测试变绿 ≠ 程序逻辑正确                       │
│                                              | 
│  绿色只说明：代码执行过程中没有抛出异常          │  ⚠️ 误导！
│  绿色不说明：运行结果符合预期                   │
│                                              │
│  ✅ 必须用 断言(Assertion) 来验证实际值与预期值│
└──────────────────────────────────────────────┘
```

### 什么是断言？

**断言 = 预期值 vs 实际值的比较判断**

```java
@Test
// ✅ 有断言 —— 结果不对时会变红，真正验证了正确性
public void testAddWithAssertion() {
    int result = Calculator.add(1, 2);       // 实际值
    assertEquals(3, result);                  // 预期值 vs 实际值
    // 如果 result != 3 → 抛出 AssertionError → 变红 ❌
    // 如果 result == 3 → 通过 → 变绿 ✅
}
```

### JUnit 常用断言方法

| 方法 | 说明 | 示例 |
|:---|:---|:---|
| `assertEquals(预期, 实际)` | 判断是否相等 | `assertEquals(3, result)` |
| `assertNotEquals(预期, 实际)` | 判断不相等 | `assertNotEquals(0, count)` |
| `assertTrue(条件)` | 判断为 true | `assertTrue(result > 0)` |
| `assertFalse(条件)` | 判断为 false | `assertFalse(list.isEmpty())` |
| `assertNull(对象)` | 判断为 null | `assertNull(obj)` |
| `assertNotNull(对象)` | 判断不为 null | `assertNotNull(str)` |
| `assertSame(预期, 实际)` | 引用相同（==） | `assertSame(a, b)` |
| `assertNotSame(预期, 实际)` | 引用不同 | `assertNotSame(x, y)` |

### 断言使用示例

```java
import static org.junit.jupiter.api.Assertions.*;

class CalculatorTest {

    @Test
    public void testAdd() {
        int result = Calculator.add(2, 3);
        assertEquals(5, result);                          // 判等
    }

    @Test
    public void testDivide() {
        double result = Calculator.divide(10, 3);
        assertEquals(3.33, result, 0.01);                 // 浮点数比较，允许误差0.01
    }

    @Test
    public void testListOperation() {
        List<String> list = new ArrayList<>();
        list.add("Hello");
        
        assertFalse(list.isEmpty());                      // 不为空
        assertEquals(1, list.size());                     // 大小为1
        assertNotNull(list.get(0));                       // 元素不为null
    }

    @Test // 带失败提示信息的断言
    public void testWithMessage() {
        int age = -1;
        assertTrue(age >= 0, "年龄不能为负数");             // 失败时显示自定义消息
    }
}
```

### 总结：没有断言的测试是无效的

```
┌──────────────────┐     ┌──────────────────┐
│   无断言的测试     │     │   有断言的测试     │
├──────────────────┤     ├──────────────────┤
│ 🟢 永远绿色       │     │ 🟢 结果正确→绿色   │
│ ⚠️ 无法发现bug     │     │ 🔴 结果错误→红色   │
│ ❌ 形同虚设       │     │ ✅ 真正有效的测试   │
└──────────────────┘     └──────────────────┘

核心原则：每个 @Test 方法至少包含一个断言语句！
```

> **一句话总结**：绿色只代表"没报错"，断言才是"验对错"。写测试不加断言，等于没写测试。



## 六、JUnit 常见注解


| `@BeforeEach` | 修饰实例方法，**每个**测试方法执行前运行一次 | 初始化资源（准备工作） |
| `@AfterEach` | 修饰实例方法，**每个**测试方法执行后运行一次 | 释放资源（清理工作） |
| `@BeforeAll` | 修饰静态方法，**所有**测试方法执行前仅运行一次 | 初始化全局资源 |
| `@AfterAll` | 修饰静态方法，**所有**测试方法执行后仅运行一次 | 释放全局资源 |
| `@ParameterizedTest` | 参数化测试，让单个测试用不同参数运行多次 | 使用后无需 `@Test` |
| `@ValueSource` | 指定参数化测试的参数来源 | 配合 `@ParameterizedTest` 使用 |
| `@DisplayName` | 自定义测试类或方法的显示名称 | 默认显示类名/方法名 |


### 注解执行示例代码

```java
public class UserServiceTest {

    @BeforeEach
    public void testBefore(){
        System.out.println("before...");
    }

    @AfterEach
    public void testAfter(){
        System.out.println("after...");
    }

    @BeforeAll //该方法必须被static修饰
    public static void testBeforeAll(){ 
        System.out.println("before all ...");
    }

    @AfterAll //该方法必须被static修饰
    public static void testAfterAll(){
        System.out.println("after all...");
    }

    @Test
    public void testGetAge(){
        Integer age = new UserService().getAge("110002200505091218");
        System.out.println(age);
    }
    
    @Test
    public void testGetGender(){
        String gender = new UserService().getGender("612429198904201611");
        System.out.println(gender);
    }
 }   
```
输出结果:
`before all ...`
`before...`
`20`
`after...`
`before...`
`男`
`after...`
`after all...`


`@ParameterizedTest`  ，`@ValueSource` ，`@DisplayName` 注解：

         System.out.println(gender);
    }
}

## 七、Maven 依赖范围（scope）

### 为什么需要依赖范围？

默认情况下，通过 Maven 引入的依赖 jar 包**可以在任何地方使用**——无论在 `src/main/` 目录下编写主程序代码，还是在 `src/test/` 目录下编写测试代码，都能正常访问。

但在实际开发中，有些依赖我们**只希望它在特定场景下生效**，例如：

- JUnit 只在测试时需要，不应该打进最终发布的 jar 包里；
- Servlet API 只在编译时需要，运行时由 Tomcat 容器提供；
- JDBC 驱动实现只在运行时需要，编译主代码时用不到。

为此，Maven 提供了 `<scope>` 标签，用于**限制依赖的作用范围**。

### scope 的取值与作用范围

| scope 取值 | 主程序编译（main） | 测试程序编译（test） | 是否参与打包 | 典型场景 |
|:---:|:---:|:---:|:---:|:---|
| **compile**（默认） | ✅ 可用 | ✅ 可用 | ✅ 打包 | 通用依赖，如 Spring 核心包 |
| **test** | ❌ 不可用 | ✅ 可用 | ❌ 不打包 | JUnit、Mockito 等测试框架 |
| **provided** | ✅ 可用 | ✅ 可用 | ❌ 不打包 | Servlet API、Lombok，运行环境已提供 |
| **runtime** | ❌ 不可用 | ✅ 可用 | ✅ 打包 | JDBC 驱动实现，编译时只需接口 |

### 使用示例

```xml
<dependencies>
    <!-- compile（默认）：全范围可用，参与打包 -->
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-context</artifactId>
        <version>5.3.27</version>
    </dependency>

    <!-- test：仅测试时可用，不参与打包 -->
    <dependency>
        <groupId>org.junit.jupiter</groupId>
        <artifactId>junit-jupiter</artifactId>
        <version>5.9.1</version>
        <scope>test</scope>
    </dependency>

    <!-- provided：编译和测试可用，打包时不包含 -->
    <dependency>
        <groupId>javax.servlet</groupId>
        <artifactId>javax.servlet-api</artifactId>
        <version>4.0.1</version>
        <scope>provided</scope>
    </dependency>

    <!-- runtime：运行时可用，主程序编译时不可用 -->
    <dependency>
        <groupId>mysql</groupId>
        <artifactId>mysql-connector-java</artifactId>
        <version>8.0.33</version>
        <scope>runtime</scope>
    </dependency>
</dependencies>
```

### 选值建议

```
┌──────────────────────────────────────────────────────────┐
│  该依赖在哪些阶段需要？                                    │
├──────────────────────────────────────────────────────────┤
│  main + test + 打包     → compile（默认，省略即可）       │
│  仅 test                → test                            │
│  main + test，运行环境提供 → provided                     │
│  test + 运行时，编译不需要 → runtime                      │
└──────────────────────────────────────────────────────────┘
```

> **一句话总结**：`<scope>` 决定了依赖"在哪儿能用、要不要打包"。合理设置 scope 可以减小最终产物体积、避免与运行环境的依赖冲突。
